Why Single-Threaded?

Node.js runs your JavaScript on one thread. But it handles thousands of concurrent connections. How?

What runs your JS code?
V8 Engine — compiles JS to machine code on a single thread
What handles I/O?
libuv — delegates I/O to OS kernel or thread pool
What connects them?
The Event Loop — picks up completed I/O and runs callbacks
Thread Pool size?
Default: 4 threads (UV_THREADPOOL_SIZE) — for fs, dns, crypto

The 6 Phases

The event loop cycles through these phases in order. Click each step to explore.

1

Timers

Executes callbacks from setTimeout() and setInterval() whose threshold has elapsed.

Key insight: Timer callbacks are not guaranteed to run at exact time — they run as soon as possible after the delay, once the event loop reaches this phase.
2

Pending Callbacks

Executes I/O callbacks deferred from the previous loop iteration (e.g., TCP errors like ECONNREFUSED).

3

Idle / Prepare

Internal use only. Node uses this for housekeeping. You won't interact with this phase directly.

4

Poll

The most important phase. Retrieves new I/O events and executes their callbacks (file reads, network responses, etc.).

Blocking behavior: If no timers are scheduled, the poll phase will wait for callbacks to be added, then execute them immediately.
5

Check

Executes setImmediate() callbacks. This runs right after the poll phase completes.

6

Close Callbacks

Handles close events like socket.on('close', ...). Cleanup happens here.

Predict the Output

Run this code and see the order. Can you explain why it prints in this order?

script.js
console.log('1 - Start'); setTimeout(() => console.log('2 - setTimeout'), 0); Promise.resolve().then(() => console.log('3 - Promise')); console.log('4 - End');
Why this order? Microtasks (Promises) run between each phase of the event loop — before the next macrotask (setTimeout). So: sync code first → microtasks → macrotasks.

setTimeout(0) vs setImmediate()

These two are often confused. Try the playground to see the non-deterministic behavior.

timing.js
// Run this multiple times — the order may change! // In Node.js, outside I/O, the order is non-deterministic setTimeout(() => console.log('setTimeout'), 0); // setImmediate is Node-only, simulated here: Promise.resolve().then(() => { console.log('setImmediate (simulated via microtask)'); }); // But INSIDE an I/O callback, setImmediate always runs first: // fs.readFile('file', () => { // setTimeout(() => console.log('timeout'), 0); // setImmediate(() => console.log('immediate')); // always first // });

process.nextTick() — The Queue Jumper

process.nextTick() runs before any other microtask. It can even starve the event loop.

nexttick.js
// Simulating process.nextTick priority // In real Node.js, nextTick runs before Promise.then const results = []; // Simulating event loop priority Promise.resolve().then(() => results.push('Promise 1')); Promise.resolve().then(() => results.push('Promise 2')); // nextTick would run BEFORE these promises in real Node.js // It has its own queue that drains before the microtask queue queueMicrotask(() => results.push('queueMicrotask')); setTimeout(() => { results.push('setTimeout'); console.log('Execution order:'); results.forEach((r, i) => console.log((i+1) + '. ' + r)); }, 10);
Danger: Recursive process.nextTick() calls will starve I/O — the event loop never reaches the poll phase. Use setImmediate() for recursive async patterns instead.
